# Token Analytics & Cost Control

Monitor LLM token usage, reduce costs with smart output compression, and optimize scanner output.

---

## Overview

In this tutorial, you'll learn to:

- Track token usage and costs across providers and commands
- Use smart output compression to cut LLM costs by 70-90%
- Leverage log deduplication to eliminate redundant CI output
- Request structured JSON from scanners for more reliable parsing

**Time estimate:** ~10 minutes

**Prerequisites:**

- DojOps installed (`npm i -g @dojops/cli`)
- A configured LLM provider
- A project initialized with `dojops init`

## Part 2: Smart Output Compression

### Step 5: Understand the Problem

When DojOps sends scanner output or CI logs to the LLM for analysis, raw output often contains:

- ANSI color codes (invisible but consume tokens)
- Hundreds of success lines ("PASS", "ok", "✓")
- Duplicated warnings and errors
- Verbose stack traces

This wastes 70-90% of tokens on information the LLM doesn't need.

### Step 6: Compression in Action

Smart compression activates automatically in `debug-ci`, `scan --fix`, and verification steps. Compare the difference:

```bash
# Without compression: ~8,000 tokens of raw CI output
dojops --debug-ci "$(cat ci-log-raw.txt)"

# With compression (automatic): ~1,200 tokens sent to LLM
# Same command — compression happens internally
```

The compression pipeline:

1. **Strip ANSI** — Remove all color/formatting escape codes
2. **Remove success lines** — Filter out lines matching pass/ok/success patterns
3. **Deduplicate** — Collapse repeated lines into `(message) ×N` format
4. **Truncate** — Keep the most relevant sections within token budget

### Step 7: See Compression Stats

When compression activates, DojOps reports the savings:

```bash
dojops --debug-ci "ERROR: npm test failed with exit code 1..."
```

```
┌  CI Debug Analysis
│
│  Output compression:
│    Raw: 12,847 lines (45,200 tokens)
│    Compressed: 234 lines (1,890 tokens)
│    Savings: 95.8%
│
│  Analysis:
│  ...
│
└  Diagnosis complete
```

## Part 3: Log Deduplication

### Step 8: Deduplication for CI Logs

Real CI logs often contain massive repetition. DojOps collapses these before LLM analysis:

```bash
# A CI log with 47 identical deprecation warnings becomes:
# "WARNING: deprecated API version (×47)"
```

This is especially valuable for `debug-ci` where CI providers output thousands of lines:

```bash
dojops --debug-ci "$(gh run view 12345 --log-failed)"
```

```
┌  CI Debug Analysis
│
│  Log deduplication:
│    Collapsed 47 duplicate warnings
│    Collapsed 12 repeated "npm WARN" lines
│    Collapsed 8 identical test timeouts
```

## Part 4: Structured JSON from Scanners

### Step 9: JSON-Native Scanner Output

When scanners support JSON output, DojOps requests it directly instead of parsing human-readable text:

```bash
dojops scan --json
```

```
┌  Security Scan (JSON mode)
│
│  Scanner output format:
│    trivy         → --format json ✓
│    checkov       → -o json ✓
│    npm-audit     → --json ✓
│    gitleaks      → --report-format json ✓
│    hadolint      → -f json ✓
│
│  Benefits:
│    More reliable parsing
│    Better compression ratios
│    Structured data for LLM analysis
│
└  5/5 scanners using JSON output
```

When combined with `--fix`, structured JSON gives the LLM cleaner input for generating targeted remediation:

```bash
dojops scan --json --fix
```

## Cost Optimization Strategies

Combine these features for maximum cost efficiency:

| Strategy                                      | Savings   | Command                                    |
| --------------------------------------------- | --------- | ------------------------------------------ |
| Use `--model fast` for simple tasks           | 60-80%    | `dojops --model fast "Create .gitignore"`  |
| Use `--thinking low` for straightforward work | 30-50%    | `dojops --thinking low "Add health check"` |
| Smart compression on CI debug                 | 70-95%    | `dojops --debug-ci` (automatic)            |
| JSON scanner output                           | 20-40%    | `dojops scan --json --fix`                 |
| Token monitoring                              | Awareness | `dojops tokens --graph`                    |
