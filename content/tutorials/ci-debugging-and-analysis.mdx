# CI Debugging & Infrastructure Analysis

Diagnose CI/CD failures, analyze infrastructure diffs, and auto-fix quality issues with DojOps.

---

## Overview

In this tutorial, you'll learn to use DojOps diagnostic commands to:

- Debug CI/CD log failures with AI-powered root cause analysis
- Analyze Terraform plan diffs for risk, cost, and security impact
- Run DevOps quality checks and auto-remediate findings
- Test LLM provider connectivity

**Time estimate:** ~15 minutes

**Prerequisites:**

- DojOps installed and configured with an LLM provider
- A project with DevOps configs (run `dojops init` first)

---

## Step 1: Debug a CI Failure

When a CI pipeline fails, paste the error log into `dojops debug ci` for AI-powered diagnosis:

```bash
dojops debug ci "npm ERR! ERESOLVE unable to resolve dependency tree
npm ERR! While resolving: my-app@1.0.0
npm ERR! Found: react@18.2.0
npm ERR! Could not resolve dependency:
npm ERR! peer react@'^17.0.0' from react-beautiful-dnd@13.1.1"
```

```
┌  CI Failure Diagnosis
│
│  Error Type: Dependency Resolution
│
│  Root Cause:
│  react-beautiful-dnd@13.1.1 requires React 17 but your project
│  uses React 18. The peer dependency constraint is incompatible.
│
│  Suggested Fixes:
│  1. Replace react-beautiful-dnd with @hello-pangea/dnd (React 18 fork)
│  2. Use --legacy-peer-deps in your npm install step
│  3. Pin react-beautiful-dnd to a version supporting React 18
│
│  CI Fix:
│  Add to your workflow: npm ci --legacy-peer-deps
│  (Not recommended long-term — migrate to @hello-pangea/dnd)
│
└  Diagnosis complete
```

Works with any CI log — GitHub Actions, GitLab CI, Jenkins, CircleCI, etc:

```bash
# TypeScript compilation failure
dojops debug ci "ERROR: tsc failed with exit code 1
src/index.ts(42,5): error TS2345: Argument of type 'string' is not assignable to parameter of type 'number'."

# Docker build failure
dojops debug ci "Step 5/12 : RUN pip install -r requirements.txt
ERROR: Could not find a version that satisfies the requirement torch==2.1.0"
```

## Step 2: Analyze an Infrastructure Diff

When running `terraform plan` or similar tools, save the output to a file and feed it to DojOps for risk analysis:

```bash
# Save the plan output to a file (recommended for multiline diffs)
terraform plan -no-color > plan.diff

# Analyze the diff
dojops analyze diff --file plan.diff
```

You can also pipe directly:

```bash
terraform plan -no-color | dojops analyze diff
```

```
┌  Infrastructure Diff Analysis
│
│  Risk Level: HIGH
│
│  Security:
│  ⚠ CRITICAL: Security group ingress changed from private (10.0.0.0/8)
│    to public (0.0.0.0/0). This exposes port 443 to the entire internet.
│
│  Cost Impact:
│  ↓ aws_instance.app destroyed — saves ~$30/month (t3.medium)
│
│  Operational:
│  ⚠ Instance destruction is irreversible. Ensure data is backed up.
│
│  Recommendations:
│  1. Restrict ingress to specific IPs or use a WAF
│  2. Verify instance destruction is intentional
│  3. Add lifecycle { prevent_destroy = true } for critical resources
│
└  Analysis complete
```

## Step 3: Run a DevOps Quality Check

Analyze your project's DevOps configuration for quality, security, and best practices:

```bash
dojops check
```

```
┌  DevOps Quality Check
│
│  Score: 72/100
│
│  Findings:
│  ├  HIGH: Dockerfile uses latest tag — pin to specific version
│  ├  HIGH: No dependency caching in GitHub Actions workflow
│  ├  MEDIUM: Missing health check in docker-compose.yml
│  ├  MEDIUM: No artifact upload step in CI pipeline
│  └  LOW: No .dockerignore file detected
│
│  Missing Configs:
│  ├  No Kubernetes manifests
│  └  No monitoring configuration
│
└  Score: 72/100
```

Get machine-readable output for CI integration:

```bash
dojops check --output json
```

```json
{
  "score": 72,
  "findings": [
    { "severity": "HIGH", "message": "Dockerfile uses latest tag", "file": "Dockerfile" },
    {
      "severity": "HIGH",
      "message": "No dependency caching in CI workflow",
      "file": ".github/workflows/ci.yml"
    }
  ],
  "missingConfigs": ["kubernetes", "monitoring"]
}
```

## Step 4: Auto-Remediate Findings

Let DojOps automatically fix HIGH and CRITICAL findings:

```bash
dojops check --fix
```

```
┌  DevOps Quality Check — Auto-Remediation
│
│  Score: 72/100
│
│  Remediating HIGH findings...
│
│  ✓ Fixed: Dockerfile — pinned base image to node:20.11-slim
│    Created backup: Dockerfile.bak
│
│  ✓ Fixed: .github/workflows/ci.yml — added dependency caching step
│    Created backup: .github/workflows/ci.yml.bak
│
│  Skipped: MEDIUM findings (use --fix with lower threshold for these)
│
│  New Score: 85/100 (+13)
│
└  2 files remediated
```

Auto-approve without interactive confirmation:

```bash
dojops check --fix --yes
```

## Step 6: Explain Previous Plans

Use `dojops explain` to get a plain-language explanation of any saved plan:

```bash
# Explain the most recent plan
dojops explain

# Explain a specific plan by ID
dojops explain plan-a1b2c3d4
```

```
┌  Plan Explanation
│
│  This plan sets up a CI/CD pipeline for a Node.js application.
│
│  It creates 3 files:
│  1. A GitHub Actions workflow that runs tests, linting, and builds
│     on every push and pull request to the main branch.
│  2. A Dockerfile with a multi-stage build for production deployment.
│  3. A docker-compose.yml for local development with hot reloading.
│
│  The tasks execute in order: CI workflow first (no dependencies),
│  then Dockerfile, then docker-compose (depends on Dockerfile).
│
└  3 tasks, estimated 2 files created + 1 updated
```
