# Observability & Learning

Discover actionable insights, learn from recurring failures, and build persistent memory across sessions.

---

## Overview

In this tutorial, you'll learn to:

- Detect optimization opportunities from your execution history
- Mine error patterns and auto-generate rules for future runs
- Use cross-session memory to retain project context and decisions
- Build a feedback loop that makes DojOps smarter over time

**Time estimate:** ~12 minutes

**Prerequisites:**

- DojOps installed (`npm i -g @dojops/cli`)
- A project initialized with `dojops init`
- Some prior scan/execution history (run a few commands first)

---

## Part 1: Opportunity Detection

### Step 1: Run Opportunity Analysis

After you've been using DojOps for a while, analyze your execution history for actionable suggestions:

```bash
dojops insights
```

```
┌  Opportunity Detection
│
│  Analyzed: 47 executions, 12 scans, 5 plans
│
│  Opportunities found: 4
│
│  1. Unused --fix flag
│     You ran `scan` 8 times without using `--fix`.
│     Auto-remediation could have resolved 23 findings automatically.
│     → Try: dojops scan --fix
│
│  2. Repeated kubectl dry-run failures
│     Kubernetes configs failed dry-run validation on the same
│     issue (missing resource limits) in 4 of 5 executions.
│     → Consider adding default resource limits to your templates.
│
│  3. Model cost optimization
│     83% of your generations used gpt-4o, but 60% were simple
│     configs (Dockerfiles, .gitignore) that gpt-4o-mini handles well.
│     → Try: dojops config alias fast gpt-4o-mini
│
│  4. Scan coverage gap
│     You have Terraform configs but never ran `scan --iac`.
│     Checkov could catch misconfigurations before apply.
│     → Try: dojops scan --iac
│
└  4 opportunities identified
```

### Step 2: Focus on Specific Areas

Filter insights by category:

```bash
# Security-focused insights
dojops insights --security

# Cost-focused insights
dojops insights --cost

# Workflow optimization
dojops insights --workflow
```

### Step 3: Act on Recommendations

Each insight includes a concrete command you can run immediately:

```bash
# Apply the suggestions
dojops scan --fix                    # Address unused --fix suggestion
dojops config alias fast gpt-4o-mini # Address cost optimization
dojops scan --iac                    # Address coverage gap
```

After acting on suggestions, `dojops insights` will update to reflect the improved state.

## Part 2: Error Pattern Learning

### Step 4: Mine Patterns from History

DojOps can analyze your execution logs and scan history to identify recurring failures:

```bash
dojops learn
```

```
┌  Error Pattern Learning
│
│  Analyzed:
│    Execution logs: 47 entries
│    Scan history: 12 scans
│    Plan results: 5 plans
│
│  Patterns detected: 3
│
│  1. Recurring CVE (scan)
│     Trivy finds CVE-2024-1234 in 4 of 5 scans.
│     Package: lodash < 4.17.21
│     → Rule: Flag lodash version in future scans
│
│  2. Terraform init failure (execution)
│     `terraform init` fails on S3 backend 3 of 4 times.
│     Root cause: Missing AWS_REGION environment variable
│     → Rule: Pre-check AWS_REGION before Terraform tasks
│
│  3. Dockerfile lint repeat (scan)
│     hadolint DL3008 (pin apt versions) in every scan.
│     Same 2 Dockerfiles never get fixed.
│     → Rule: Prioritize DL3008 in remediation suggestions
│
└  3 patterns → rules saved to .dojops/rules.md
```

### Step 5: Understand the Rules File

The learned patterns are saved as rules that DojOps injects into future LLM prompts:

```bash
cat .dojops/rules.md
```

```markdown
# Learned Rules (auto-generated by `dojops learn`)

## Scan Patterns

- lodash < 4.17.21 is a recurring CVE — suggest upgrade early
- hadolint DL3008 appears in every scan — prioritize this fix

## Execution Patterns

- Terraform tasks in this project require AWS_REGION — verify before init
```

These rules are automatically included in LLM context for `scan --fix`, `debug-ci`, and `plan` commands, so the AI learns from your project's specific history.

### Step 6: Manage Rules

Add manual rules based on your team's knowledge:

```bash
# Rules file is plain markdown — edit directly
echo "- Always use Node 20 LTS in CI workflows" >> .dojops/rules.md
```

Reset learned rules to start fresh:

```bash
dojops learn --reset
```

Re-learn from current history:

```bash
dojops learn --refresh
```

## Part 3: Memory / RAG Across Sessions

### Step 7: Understand Cross-Session Memory

DojOps uses vector embeddings (LanceDB) to remember context across `dojops chat` sessions. It remembers:

- Past configurations you generated
- Infrastructure decisions you made
- User preferences for this project
- Patterns from previous conversations

### Step 8: Use Memory in Chat

Start a chat session and DojOps will recall past context:

```bash
dojops chat
```

```
┌  DojOps Chat (memory enabled)
│
│  💡 Recalled context from 3 previous sessions:
│    - Last Terraform setup used S3 backend with DynamoDB locking
│    - Project uses GitHub Actions with Node 20/22 matrix
│    - Preferred Kubernetes namespace: "production"
│
◆  How can I help?
│
│  > Set up a new Terraform module for RDS
│
│  Based on your previous Terraform setup, I'll configure this
│  with the same S3 backend and DynamoDB locking pattern you
│  used before...
```

Memory means you don't have to repeat context. DojOps remembers your project's conventions.

### Step 9: Memory in Subsequent Sessions

Close and reopen chat — memory persists:

```bash
# Session 1
dojops chat
> "We use Terraform modules from our internal registry at registry.company.com"
# ... chat continues ...
# (exit)

# Session 2 (days later)
dojops chat
> "Create a new VPC module"
# DojOps automatically uses your internal registry URL
```

```
┌  DojOps Chat
│
│  Creating VPC module with source from registry.company.com
│  (recalled from session on Mar 10)
│
│  ...
```

### Step 10: View Memory Status

Check what DojOps has memorized for the current project:

If you want to start fresh:


## The Learning Feedback Loop

These three features create a continuous improvement cycle:

```
  Run commands
       │
       ▼
  Build history ──→ dojops insights (spot opportunities)
       │                    │
       ▼                    ▼
  dojops learn ──→ .dojops/rules.md (auto-inject into LLM)
       │
       ▼
  dojops chat ──→ memory (remember decisions across sessions)
       │
       ▼
  Better results next time
```

The more you use DojOps, the smarter it gets for your specific project.
