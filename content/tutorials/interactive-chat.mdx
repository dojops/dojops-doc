# Interactive chat sessions

Use DojOps chat for multi-turn AI conversations with agent routing, slash commands, and session persistence.

> **Difficulty:** Beginner
> **Duration:** 25 minutes
> **What you'll build:** A working understanding of the DojOps chat interface вҖ” you'll run scans, generate infrastructure configs, and switch between specialist agents, all from within a single persistent chat session.

---

## What you'll learn

- How the chat session maintains context across multiple turns
- Which slash commands are available and when to use each one
- How agent routing works and how to override it manually
- How named sessions persist across terminal restarts
- How to use single-message mode for scripting and CI/CD integration

---

## Prerequisites

- DojOps installed: `npm i -g @dojops/cli`
- An API key configured (`dojops provider add <provider> --token <token>`)
- A project directory with `dojops init` already run (run it on any project, or use the [example repo](https://github.com/dojops/example-nextjs.git))

## Workshop steps

### Step 1: Start your first chat session

Launch the interactive session:

```bash
dojops chat
```

```
в”Ң  DojOps Chat
в”Ӯ  Session: chat-a1b2c3
в”Ӯ  Agent: auto-routing (32 specialists available)
в”Ӯ
в—Ҷ  You: _
```

The session ID is assigned automatically. "Auto-routing" means DojOps analyzes each message and routes it to the most appropriate specialist from the 32 built-in agents. You don't need to specify which agent to use вҖ” but you can override it when you want to.

Ask your first question:

```
в—Ҷ  You: What CI/CD tools are configured in this project?
в”Ӯ
в—Ү  DojOps [github-actions-specialist]:
в”Ӯ  Based on the project context at .dojops/context.json, I can see:
в”Ӯ
в”Ӯ  - GitHub Actions workflow at .github/workflows/ci.yml
в”Ӯ    Triggers: push and pull_request on main
в”Ӯ    Steps: checkout, setup-node, install, build, test
в”Ӯ    Missing: dependency caching, security audit step
в”Ӯ
в”Ӯ  - Dockerfile (multi-stage build)
в”Ӯ    Base: node:20-slim (unpinned вҖ” consider pinning the patch version)
в”Ӯ
в”Ӯ  No Terraform, Kubernetes, or Docker Compose detected.
в”Ӯ
в—Ҷ  You: _
```

The bracket annotation вҖ” `[github-actions-specialist]` вҖ” shows which agent handled your message. The response references your actual project context, not generic advice. Ask a follow-up:

```
в—Ҷ  You: What's missing from the CI workflow?
в”Ӯ
в—Ү  DojOps [github-actions-specialist]:
в”Ӯ  The workflow covers the basics but has two gaps worth fixing now:
в”Ӯ
в”Ӯ  1. No dependency caching вҖ” every run reinstalls node_modules from
в”Ӯ     scratch. actions/cache for node_modules typically saves 30-60s.
в”Ӯ
в”Ӯ  2. No security step вҖ” npm audit isn't running, so vulnerable
в”Ӯ     dependencies could reach production undetected.
в”Ӯ
в”Ӯ  Both are quick adds. Want me to generate an improved workflow?
в”Ӯ
в—Ҷ  You: _
```

The conversation history carries across turns. The agent knows this is a follow-up about the same CI workflow from the previous message.

Type `/exit` to end the session at any time.

### Step 2: Use slash commands

Slash commands let you trigger DojOps operations from within the chat without leaving the session. Here's the full set:

| Command              | What it does                                              |
| -------------------- | --------------------------------------------------------- |
| `/help`              | Show all available commands                               |
| `/exit`              | Save session and exit                                     |
| `/agent <name>`      | Pin to a specialist agent (`auto` to unpin)               |
| `/model`             | Switch LLM model (interactive picker)                     |
| `/provider [name]`   | Switch LLM provider mid-session                           |
| `/compress`          | Summarize conversation to free context window             |
| `/sessions`          | List saved chat sessions                                  |
| `/status`            | Show session status bar (agent, provider, turn count)     |
| `/history`           | Show recent messages in this session                      |
| `/clear`             | Clear conversation history (keeps session)                |
| `/save`              | Save the current session to disk                          |
| `/plan <goal>`       | Decompose a goal into a task graph                        |
| `/apply [plan-id]`   | Execute a saved plan                                      |
| `/scan [type]`       | Run a security scan and show results inline               |
| `/auto <prompt>`     | Run autonomous agent (iterative tool-use)                 |
| `/checkpoint [name]` | Create a git-based snapshot of current state              |
| `/restore <id>`      | Restore files from a checkpoint                           |
| `/rewind [n]`        | Undo last N conversation turns (`--code` to revert files) |
| `/voice`             | Push-to-talk voice input (requires whisper.cpp and sox)   |
| `!<command>`         | Run a shell command inline (e.g. `!git status`)           |
| `@path/to/file`      | Inject file contents into your message as context         |

### Step 3: Pin to a specialist agent

Sometimes you want to stay focused on a specific domain for an entire session. Use `--agent` at startup to pin:

```bash
dojops chat --agent=terraform
```

```
в”Ң  DojOps Chat
в”Ӯ  Session: chat-b2c3d4
в”Ӯ  Agent: terraform-specialist (pinned)
в”Ӯ
в—Ҷ  You: _
```

Every message in this session goes to `terraform-specialist`, regardless of topic keywords:

````
в—Ҷ  You: Create an S3 bucket with versioning and encryption enabled
в”Ӯ
в—Ү  DojOps [terraform-specialist]:
в”Ӯ  Here's a Terraform configuration for an S3 bucket with versioning
в”Ӯ  and server-side encryption:
в”Ӯ
в”Ӯ  ```hcl
в”Ӯ  resource "aws_s3_bucket" "main" {
в”Ӯ    bucket = "${var.project_name}-${var.environment}-data"
в”Ӯ  }
в”Ӯ
в”Ӯ  resource "aws_s3_bucket_versioning" "main" {
в”Ӯ    bucket = aws_s3_bucket.main.id
в”Ӯ    versioning_configuration {
в”Ӯ      status = "Enabled"
в”Ӯ    }
в”Ӯ  }
в”Ӯ
в”Ӯ  resource "aws_s3_bucket_server_side_encryption_configuration" "main" {
в”Ӯ    bucket = aws_s3_bucket.main.id
в”Ӯ    rule {
в”Ӯ      apply_server_side_encryption_by_default {
в”Ӯ        sse_algorithm = "AES256"

### Step 4: Named sessions and resume

Named sessions let you maintain context across multiple terminal sessions. Start one with `--session`:

```bash
dojops chat --session=infra-migration
```

```
в”Ң  DojOps Chat
в”Ӯ  Session: infra-migration
в”Ӯ  Agent: auto-routing
в”Ӯ
в—Ҷ  You: We're migrating from EC2 to ECS. Let's plan the Terraform changes needed.
в”Ӯ
в—Ү  DojOps [terraform-specialist]:
в”Ӯ  For an EC2 to ECS migration, you'll need to create or modify:
в”Ӯ
в”Ӯ  1. aws_ecs_cluster вҖ” The ECS cluster replacing your EC2 Auto Scaling group
в”Ӯ  2. aws_ecs_task_definition вҖ” Container definition (CPU, memory, image)
в”Ӯ  3. aws_ecs_service вҖ” Service with desired count and load balancer attachment
в”Ӯ  4. IAM roles вҖ” Task execution role and task role
в”Ӯ
в”Ӯ  What's your current EC2 setup? AMI-based, or do you have a Packer pipeline?
в”Ӯ
в—Ҷ  You: AMI-based, t3.medium, behind an ALB
в”Ӯ
в—Ү  DojOps [terraform-specialist]:
в”Ӯ  Good. In that case the ALB can stay вҖ” you just swap the target group
в”Ӯ  target type from instance to ip...
в”Ӯ
в—Ҷ  You: /save
в”Ӯ
в—Ү  DojOps: Session saved: infra-migration (4 messages)
в”Ӯ
в—Ҷ  You: /exit
```

Close your terminal, come back tomorrow, and resume exactly where you left off:

```bash
dojops chat --session=infra-migration
```

```
в”Ң  DojOps Chat
в”Ӯ  Session: infra-migration (resumed)
в”Ӯ  Messages loaded: 4
в”Ӯ
в—Ҷ  You: Let's continue. What about the security groups?
в”Ӯ
в—Ү  DojOps [terraform-specialist]:
в”Ӯ  Right, picking up from the ALB target group. For security groups,
в”Ӯ  you'll need to update the ECS task security group to accept traffic
в”Ӯ  from the ALB security group on port 3000 (or whatever your app port is)...
```

The agent has the full conversation history. You don't need to re-explain your EC2 setup.

Resume the most recent session without knowing the name:

```bash
dojops chat --resume
```

### Step 5: Single-message mode

Use `--message` (or `-m`) to send one message and get the response without entering interactive mode. This is how you integrate DojOps into scripts and CI pipelines:

```bash
dojops chat --message "What DevOps files are missing from this project?"
```

```
Based on .dojops/context.json, the following are missing:

- Docker Compose file (no docker-compose.yml detected)
- Infrastructure as Code (no Terraform or Pulumi configs)
- Container orchestration (no Kubernetes manifests)
- Monitoring configuration (no Prometheus or Grafana configs)
- Dependency update automation (no Dependabot or Renovate config)
```

Get structured JSON output for downstream processing:

```bash
dojops chat -m "Summarize the project's DevOps maturity" --output json
```

```json
{
  "response": "The project has GitHub Actions CI and a Dockerfile but lacks IaC, container orchestration, and monitoring. DevOps maturity: Basic (estimated 60-65/100).",
  "agent": "devops-generalist",
  "session": "msg-c3d4e5",
  "tokensUsed": 312
}
```

Use this in a CI step to generate a summary comment on a pull request, or to gate deployments based on maturity score.

### Step 6: Combine agent pinning with scripts

Pairing `--message` with `--agent` gives you targeted, repeatable queries you can automate:

```bash
# Ask the Terraform specialist about module structure for a new service
dojops chat --agent=terraform -m "What Terraform module structure should I use for a new microservice?"

# Ask the security specialist to prioritize the latest scan
dojops chat --agent=security -m "Analyze .dojops/scan-history/scan-latest.json and list the top 3 fixes by impact"

# Ask the Kubernetes specialist to review a manifest
dojops chat --agent=kubernetes -m "Review my deployment.yaml for best practices" --file deployment.yaml
```

These patterns work well in pre-commit hooks, pull request workflows, and any automation where you want a consistent expert opinion on a specific artifact.

## Troubleshooting

**Chat session shows "context not found" warnings**
You haven't run `dojops init` in this directory. The chat works without a context file but gives generic answers. Run `dojops init` and restart the chat вҖ” responses become project-specific.

**Agent routing picks the wrong specialist**
Auto-routing uses keyword matching with confidence scoring. If your prompt is ambiguous (e.g., "create a config file" could match several agents), pin explicitly: `dojops chat --agent=terraform` or use `/agent terraform` mid-session.

**`--resume` loads the wrong session**
`--resume` loads the most recently modified session file. If you have multiple sessions and want a specific one, use `dojops chat --session=<name>`. Run `dojops chat --list-sessions` to see all saved sessions.

**Voice input says "whisper.cpp not found"**
`/voice` requires two external tools: `whisper.cpp` for transcription and `sox` for audio recording. Install `sox` via your package manager (`brew install sox` / `apt install sox`). Follow the [whisper.cpp setup guide](https://github.com/ggerganov/whisper.cpp) for the transcription binary.
